# Fit Criteria

Score each job from 0 to 100 based on how well it matches my background. Read the job description as a whole and ask: "Would a hiring manager look at this person's background and think they could do this job?"

Use your judgment holistically. Specific tech overlap matters when listed, but equally important is whether the *type of work* matches. A role asking for "experience building modern web applications with component-based frameworks" is a strong match even if it names Vue instead of React, because the skills transfer directly.

Don't worry about "too junior" or "overqualified" — I'll decide that myself.

Full-stack roles are worth considering, but if the role expects deep backend expertise (designing distributed systems, managing databases at scale), score it 40 or lower unless the frontend component is substantial.

## Scoring guide

- **90-100 — Strong fit.** The role describes work I've directly done. React and TypeScript are central. A recruiter would see my background and immediately want to talk.
- **70-89 — Good fit.** Most of the role aligns with my experience. Maybe a different frontend framework (Vue, Angular) or some backend expectations I could ramp into. A recruiter would assume I could do the work with a quick adjustment.
- **50-69 — Reasonable fit.** Some meaningful overlap, but the role leans into areas I haven't worked in much, or the primary tech is unfamiliar. Worth applying if nothing better is available.
- **25-49 — Stretch.** The overlap is thin. I'd be learning on the job for most of it.
- **0-24 — Poor fit.** Fundamentally different kind of work from what I've done.

## What signals a good match

- React and TypeScript as primary technologies
- Building user-facing web applications
- Component-based architecture and modern frontend tooling
- Collaboration with design teams
- Companies that care about UI/UX quality and frontend craft

## What signals a weaker match

- Backend-heavy roles where frontend is an afterthought
- Roles centered on mobile development (React Native is borderline — score case by case)
- Roles where the primary framework is something niche with low transferability (e.g., Ember, Elm)
- Pure infrastructure, data pipeline, or ML roles
